# FreeCAD 電子零件盒參數命名字典 — 版本與架構說明

## 版本資訊

| 項目 | 內容 |
|---|---|
| 檔名 | `FreeCAD_電子零件盒參數命名字典_v3.html` |
| 型態 | 單一 HTML 檔案，內含 CSS／JS，無需伺服器，瀏覽器直接開啟即可用 |
| 資料儲存 | 瀏覽器 `localStorage`，鍵名 `freecad_param_dictionary_v2` |
| 本版（v3）主要異動 | 將原本「文字提示式」的拖曳排序，換成 iOS 相片風格拖曳排序，並加入多選群組拖曳 |

### 這一版改了什麼

舊版（v2）拖曳排序時，畫面上會出現「放到這裡之前／之後」的文字標籤，以及畫面正中央浮動的「將移到第 N 位」提示泡泡。這版拿掉了這些文字提示，改成：

- 拿起項目時整張卡片輕微放大、傾斜、抬升出陰影
- 其餘項目用平滑動畫讓位（不是瞬間跳動）
- 可以先「多選」多個項目，再拖曳其中一個，全部一起移動，並顯示堆疊卡片視覺＋數量徽章
- 拖曳判斷邏輯改用「格子高度快取 ＋ 游標位置」運算，修正了快速拖曳時位置會亂跳（直接跳最上或最下）的問題

套用範圍：**命名字典清單、專案群組的小群組排序、小群組內的字卡排序（含跨小群組移動）**，三處共用同一套邏輯。

## 整體架構

```
單一 HTML 檔案
├── <style>   版面、卡片、拖曳視覺樣式
├── <body>    命名字典／專案群組 兩個分頁的 DOM 結構、各種對話框
└── <script>  狀態管理、渲染、拖曳排序、匯入匯出
```

### 資料模型（`state`）

```js
state = {
  items: [                     // 命名字典 — 所有參數命名
    { id, category, name, zh, suffix, tags[], favorite, note, active, order }
  ],
  projects: [                  // 專案群組
    {
      id, name, order,
      subgroups: [              // 小群組
        { id, name, order, cardIds: [item.id, ...] }  // 只存 id，實際內容仍取自 items
      ]
    }
  ],
  nextProjectId, nextSubgroupId
}
```

- `state.items` 是唯一的資料來源（single source of truth），專案／小群組只是用 `cardIds` 陣列參照這些字卡，不會複製資料。
- 命名字典的排序靠 `order` 數值；小群組排序、小群組內字卡順序都是靠陣列本身的順序（`subgroups` 依 `.order` 排序，`cardIds` 依陣列順序）。
- 全部狀態存進 `localStorage`；「匯出」是把整包 `state` 存成 JSON 檔，「匯入」則是讀回來蓋掉現有狀態。

### 兩個主要畫面（view）

1. **命名字典**（`#dictionaryView`）：搜尋／分類／狀態篩選、卡片清單、新增與編輯命名、指派到專案。
2. **專案群組**（`#projectsView`）：左側專案清單，右側該專案底下的小群組與字卡，可從字典挑字卡加入。

## 拖曳排序模組（本版重點）

所有拖曳共用同一組輔助函式，寫在 script 區塊「共用：相片風格＋多選堆疊拖曳排序」註解之下：

| 函式 | 作用 |
|---|---|
| `heightCache(container, selector)` | 拖曳開始那一刻，把容器內每個項目的實際高度存起來（用 id 對應），之後全程用這份快取，不再即時量測畫面 |
| `indexByPointer(container, otherIds, cache, gap, clientY)` | 用「游標在容器內的相對位置」＋ 快取高度，累加計算目前應該插入第幾個位置 |
| `mergeReorderedSubset(fullIds, visibleIdsOriginal, newVisibleSeq)` | 命名字典有篩選／搜尋時，只重排「目前顯示中」的項目，再把結果安全地合併回完整順序，不影響被篩選掉的項目 |
| `addStack(el, count)` | 幫拖曳中的項目加上堆疊卡片視覺（最多疊 2 張）＋ 右上角數量徽章 |
| `setDragVisual` / `clearDragVisual` | 套用／還原相片風格的抬升、傾斜、陰影效果 |
| `flipOthers` / `captureRects` | 用 FLIP 技巧讓其他項目「平滑補位」，而不是瞬間跳到新位置 |

### 為什麼改用這套邏輯（修正亂跳問題）

舊邏輯是每次移動滑鼠就即時去量「其他項目目前在畫面上的座標」來判斷該插入哪裡；但那些項目可能還在補位動畫途中，量到的座標會失真，快速拖曳時就直接跳最上或最下。

新邏輯拿掉了「即時量測動畫中項目」這一步：高度在拖曳開始那一刻就快取起來（之後不變），插入位置只看「游標在容器內的相對高度位置」去累加比對，兩者都不受動畫影響，因此不管拖多快都會照順序一格一格移動。這套算法同時也能正確處理**高度不一致**的項目（例如小群組，因為裡面卡片數量不同，高度本來就不一樣）。

### 三處套用方式

1. **命名字典清單**（`dictPointerMove` / `dictPointerUp`）
   排序範圍限定在「啟用中」或「已暫停」同一狀態內（維持原本行為）；若清單目前有搜尋／篩選，只會重排看得到的項目，其餘項目位置不受影響。

2. **小群組排序**（`subgroupPointerMove` / `subgroupPointerUp`）
   單一容器內排序，因為小群組高度不固定，靠 `heightCache` 處理。

3. **小群組內字卡排序**（`cardPointerMove` / `cardPointerUp`）
   除了同小群組內重排，也支援**拖到別的小群組**：拖曳中放開的小群組容器會出現虛線外框提示（`.drop-target-hilite`），放開後字卡會從原本的小群組移除、插入到新的小群組。

### 多選群組拖曳

三處都有「多選」按鈕（`multiSelectDictBtn` / `multiSelectProjectBtn`）：

- 開啟後，項目前面會出現 ☐／☑ 勾選框，可以勾選多個
- 拖曳其中任一個「已勾選」項目的把手（☰），會把所有勾選項目當成一組一起搬動
- 拖曳中的項目會疊上 1～2 張錯位卡片＋圓形數量徽章，模擬「抓著一疊卡片」的感覺
- 放開後，多選狀態會保留（跟 iOS 相片 App 一樣），方便連續操作；按一次「多選」按鈕可關閉並清除選取

## 已知限制／待留意的邊界情況

- 命名字典若在「篩選中」拖曳排序，只精確處理「同時符合啟用／暫停狀態」且「有被顯示」的項目；如果選取了不同啟用狀態的項目做多選拖曳，會自動被排除在該次群組之外。
- 目前拖曳只支援滑鼠與觸控（Pointer Events），沒有另外做鍵盤排序的替代操作。
- 多選群組拖曳沒有數量上限，但項目非常多時，因為每次挪動都會整個區塊重新渲染，理論上會比單筆拖曳略慢（一般使用情境下的字卡與小群組數量不會有感）。
